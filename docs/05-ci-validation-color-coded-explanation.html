Workflow header and triggers

Defines the workflow name and when it runs. Here it runs mainly for pull requests into staging, which makes it your official validation gate.

Permissions and global env

Lists workflow permissions and non-secret variables. The CI database values are intentionally named TEST_* so they are clearly different from real infrastructure secrets.

Temporary PostgreSQL service

Starts a short-lived PostgreSQL container inside the GitHub runner. The POSTGRES_* variable names stay unchanged because the official postgres image expects those exact names.

Checkout and Node setup

Downloads the repository code and installs the Node.js version needed by both backend and frontend jobs.

Dependency installation and linting

Installs packages in backend/frontend and performs the fast static checks first.

Backend runtime tests

Runs backend unit tests, starts the backend against the temporary CI database, waits for the health endpoint, and then runs API/integration tests.

Frontend tests and Docker validation

Runs frontend tests and validates both Dockerfiles by building the images without yet pushing them anywhere.

Cleanup and PR feedback

Stops the background backend process and, if the workflow failed on a pull request, writes a helpful comment back to the PR.
