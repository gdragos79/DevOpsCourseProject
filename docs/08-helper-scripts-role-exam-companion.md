# Helper Scripts in the CI/CD Pipeline — Exam Companion

This document explains the role of the helper scripts included in the CI/CD pack. It is written as if I were the student in front of the examining commission and had to explain not only **what** each script does, but also **why** I composed it that way.

## Why helper scripts exist at all

I did not put every command directly into the GitHub Actions YAML files. Instead, I extracted the operational parts into helper scripts for four reasons:

1. **Readability** — the workflow file should explain the pipeline stages, not hide complex SSH commands inside long YAML blocks.
2. **Reusability** — the same deployment logic can be reused manually or from another workflow.
3. **Maintainability** — changing a deployment command in one script is easier than editing multiple long inline `run:` blocks.
4. **Operational separation** — the workflow orchestrates the process, while the scripts execute the server-side actions.

That separation is important in an exam because it shows that I understand the difference between orchestration logic and runtime deployment logic.

## staging-deploy.sh

**Role in the pipeline:** used by Pipeline 2 after the backend and frontend images have already been built and pushed to GHCR.

**What it does:**
- receives a `BUILD_TAG` generated by the staging workflow
- prepares a fixed application root on the staging server
- is the place where the real `docker compose pull` and `docker compose up -d` commands will run

**Why I composed it this way:**
I kept the staging deployment in its own script because staging is the first live environment in the pipeline. I want the workflow to remain readable while the staging host keeps the host-specific deployment details. A fixed root such as `/opt/remote-print/staging` also makes the deployment repeatable and easier to document.

**Why there are placeholders:**
The pack is generic. I deliberately did not guess your exact Compose filenames, image references, or runtime flags. Leaving placeholders is safer than pretending to know values that belong to your real VM.

## deploy-target.sh

**Role in the pipeline:** used by Pipeline 3 to deploy an already-built release tag to the idle blue or green environment.

**What it does:**
- requires `TARGET_COLOR` and `RELEASE_TAG`
- resolves the correct host and user based on the color
- connects to the chosen VM over SSH
- prepares the target folder
- logs in to GHCR on the remote host
- marks the exact place where real compose pull, compose up, migration, and data-sync logic belongs

**Why I composed it this way:**
In blue/green deployment the deployment logic should stay generic and the selected color should be only an input. That is why I use one script with a color switch instead of two nearly identical scripts. This reduces duplication and makes the release logic easier to justify in an exam.

**Design justification:**
Pipeline 2 builds and tags the artifacts. Pipeline 3 does not rebuild them; it only deploys the requested tag. This is a stronger release-engineering pattern because the same tested artifact moves forward.

## remote-healthcheck.sh

**Role in the pipeline:** used by Pipeline 3 immediately after deployment to the idle environment and before the proxy switch.

**What it does:**
- resolves the idle color to a real host
- runs a backend health check from inside that target server

**Why I composed it this way:**
I want a quick smoke test before switching traffic. Running the check on the target VM itself confirms that the backend service is alive locally. This helps distinguish 'the service is not running' from 'the public routing is broken'.

**Design justification:**
A smoke test should be short and deterministic. It does not replace the full API test suite; it verifies the minimum condition required before sending user traffic to the new environment.

## switch-proxy.sh

**Role in the pipeline:** used by Pipeline 3 to make the newly deployed idle environment become the active production environment.

**What it does:**
- connects to the proxy VM
- calls a helper that sets the active color
- validates Nginx configuration with `nginx -t`
- reloads Nginx

**Why I composed it this way:**
The proxy switch is the most sensitive moment of a blue/green release. I therefore kept it in a dedicated script and inserted `nginx -t` before reload. That is a defensive choice: I prefer the switch to fail safely rather than load broken proxy configuration.

**Design justification:**
Separating the switch from the deployment makes the workflow easier to reason about. First deploy. Then smoke test. Then switch traffic. This ordering is intentional and aligns with safe-release practice.

## monitor-rollback.sh

**Role in the pipeline:** used by Pipeline 3 after the proxy switch to observe the release during the rollback window.

**What it does:**
- checks the public frontend URL and public backend health URL repeatedly
- if a failure appears, switches traffic back to the previous color
- exits with an error so the workflow records the release as failed

**Why I composed it this way:**
Not every failure appears before the traffic switch. Some issues surface only when the environment is live. That is why I added a rollback window instead of ending the pipeline immediately after reload.

**Design justification:**
The script is intentionally conservative: one unhealthy check is enough to trigger rollback. In an exam context that is easy to justify because availability is prioritized over keeping a doubtful release live.

## Why these scripts are intentionally small

I deliberately kept the scripts compact in the distributed pack. That was not because the missing lines are unimportant, but because the generic archive cannot safely invent your exact VM details. The scripts show the **correct control points** in the pipeline:

- staging deploy point
- production deploy point
- pre-switch smoke test point
- proxy switch point
- post-switch rollback point

That structure is what matters architecturally. Your real project then replaces the placeholders with your exact Compose commands, image names, paths, and migration logic.

## How I would explain the overall script strategy to the commission

> 'The GitHub Actions workflows coordinate the stages of the pipeline, but the operational actions live in helper scripts. I designed it that way so the workflows remain readable and the host-specific logic is easier to maintain, reuse, and justify. Each script corresponds to one release-engineering responsibility: deploy to staging, deploy to the idle production target, validate readiness, switch traffic, and observe the rollback window.'
